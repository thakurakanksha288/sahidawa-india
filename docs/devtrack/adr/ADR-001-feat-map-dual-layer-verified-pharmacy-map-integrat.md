# ADR — feat(map): dual-layer verified pharmacy map integration

> **Date:** 2026-05-18 | **PR:** #223 | **Status:** Accepted

## Context

The existing pharmacy map relied solely on OpenStreetMap (OSM) data, which lacked official verification, specific contact details, and a clear distinction for trusted sources. To enhance user trust, provide access to verified Jan Aushadhi Kendras (JAKs), and offer a more comprehensive view of available pharmacies, a mechanism was required to integrate and prioritize official, verified data alongside general OSM data.

## Decision

A dual-layer pharmacy map integration was implemented, fetching and displaying verified pharmacy data from the SahiDawa backend alongside general pharmacy data from OpenStreetMap.

**Implementation Details:**

*   **Backend (`apps/api`):**
    *   The `/api/pharmacies/nearest` endpoint was enhanced to return up to 200 results within a configurable radius (default 50km, max 200km) and enriched with `phone_number`, `is_verified`, `district`, and `state` fields.
    *   A new `GET /api/pharmacies/in-bounds` endpoint was added to support bounding-box queries.
    *   Shared helper functions for coordinate extraction, pharmacy formatting, and robust error handling (including Supabase configuration validation) were introduced.
*   **Frontend (`apps/web`):**
    *   **Data Fetching:** Verified pharmacy data (from SahiDawa backend) and OSM pharmacy data (via Overpass API) are fetched in parallel using `Promise.allSettled` to ensure independent operation and graceful degradation.
    *   **Data Processing:** Proximity-based deduplication (100m latitude-aware) is applied to prioritize verified stores and remove redundant OSM entries.
    *   **User Interface:**
        *   Verified pharmacies are visually distinguished with a glowing green shield marker and a "Verified Safe Store" badge in map popups and drawer cards.
        *   Pharmacies are explicitly sorted, displaying verified stores first.
        *   A "Verified Partners Only" filter toggle was added to allow users to narrow results.
        *   Status text was updated to reflect the presence of both verified and OSM data.
    *   **Security:** `escapeHtml()` utility was implemented for XSS hardening of all string interpolations within map popup HTML.
    *   **Resilience:** Frontend API client functions (`fetchVerifiedPharmacies`, `fetchVerifiedPharmaciesInBounds`) return empty arrays on backend fetch failures, allowing the map to gracefully degrade to OSM-only mode.

## Context

The SahiDawa platform aims to provide reliable health information and access to safe medicine. Previously, the map feature primarily displayed pharmacy locations sourced from OpenStreetMap (OSM), which lacked official verification, specific contact details, and a clear distinction for trusted partners. This presented a challenge in guiding users to verified Jan Aushadhi Kendras (JAKs) and building confidence in medicine sourcing. A need arose to integrate and prioritize official, verified pharmacy data from the SahiDawa backend alongside the broader, unverified OSM data to offer a more comprehensive and trustworthy user experience.

## Decision

A dual-layer pharmacy map integration was implemented, allowing the frontend to fetch and display both verified pharmacy data from the SahiDawa backend and general pharmacy data from OpenStreetMap.

**Implementation Details:**

*   **Backend (`apps/api`):**
    *   The `/api/pharmacies/nearest` endpoint was enhanced to remove the 3-result cap, now returning up to 200 pharmacies within a configurable radius (default 50km, max 200km).
    *   Response data was enriched to include `phone_number`, `is_verified`, `district`, and `state` fields.
    *   A new `GET /api/pharmacies/in-bounds?south=&west=&north=&east=` endpoint was added for bounding-box queries.
    *   Shared utility functions (`extractCoordinates`, `formatPharmacy`, `validateSupabaseConfig`, `handleFetchError`) were extracted to improve code organization and error handling.
    *   Default `is_verified` status for unknown pharmacies was set to `false` for a safe default.
*   **Frontend (`apps/web`):**
    *   **Data Fetching:** Implemented parallel fetching of verified pharmacies (from the SahiDawa backend) and OSM pharmacies (via Overpass API) using `Promise.allSettled` to ensure independent operation and resilience.
    *   **Data Processing:** Proximity-based deduplication (100m latitude-aware) was applied to remove OSM duplicates of verified stores, prioritizing the verified data.
    *   **User Interface:**
        *   Verified pharmacies are rendered with a distinct glowing green shield marker and a pulsing CSS animation.
        *   A "Verified Safe Store" green banner is displayed in map popups and pharmacy drawer cards for verified locations.
        *   Pharmacies are explicitly sorted, displaying verified stores first, then by distance within each group.
        *   A "Verified Partners Only" filter toggle with a Shield icon was added to allow users to narrow results.
        *   The map's status text was updated to indicate "Verified + OSM" when both data sources contribute results.
    *   **Security:** An `escapeHtml()` utility was implemented to sanitize all string interpolations in map popup HTML, mitigating potential Cross-Site Scripting (XSS) vulnerabilities.
    *   **Resilience:** Frontend API client functions (`fetchVerifiedPharmacies`, `fetchVerifiedPharmaciesInBounds`) were designed to return empty arrays on backend fetch failures, allowing the map to gracefully degrade to OSM-only mode without console errors.
